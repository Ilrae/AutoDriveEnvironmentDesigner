# AutoDrive Environment Designer 추가 TODO 정리

작성일: 2026-04-07

## 목적

현재 프로토타입은 `Track Stage`, `Intermediate Stage`, `Practical Stage`의 기본 검증 루프를 모두 갖춘 상태이다.  
다음 단계에서는 구조를 크게 흔들기보다는, 발표 시연과 이후 확장을 고려한 사용자 기능과 가시성을 보강하는 방향으로 진행한다.

이번 TODO 리스트는 다음 구현 후보를 정리한 것이다.

## 우선순위 TODO

### 1. 차량 모델 선택 기능
- 상태: 예정
- 우선순위: 높음
- 목표:
  - 사용자가 앱 안에서 기본 ego 차량의 blueprint를 직접 지정할 수 있게 한다.
  - 현재 기본값은 `built-in vehicle + hero role`로 유지하되, 필요하면 다른 차량으로 쉽게 바꿀 수 있도록 한다.
- 구현 방향:
  - 별도 설정창에서 `preferred blueprint id`, `role_name`을 입력받는다.
  - 메인 화면은 복잡해지지 않도록 기본값만 유지하고, 상세 설정은 팝업 창에서 다룬다.
- 완료 기준:
  - 사용자가 특정 `vehicle.*` blueprint id를 입력하고 실행했을 때 해당 차량이 실제로 스폰된다.
  - 세션 요약에도 현재 선택된 차량 설정이 표시된다.

### 2. 주행 코드 선택 / 연결 기능
- 상태: 진행 중 baseline 존재
- 우선순위: 매우 높음
- 목표:
  - 기본 수동 주행 외에 외부 주행 코드 또는 ROS bridge 기반 주행 스택을 앱에서 실행할 수 있게 한다.
  - 앱은 환경 생성 / 평가 / 결과 저장을 담당하고, 외부 코드가 실제 센서 처리와 제어를 담당하는 구조를 유지한다.
- 구현 방향:
  - 기본값은 `manual` 유지
  - 확장값으로 `external_command`를 지원
  - 외부 명령은 앱이 ego vehicle을 준비한 뒤 실행하고, 외부 스택은 `role_name`으로 ego actor를 찾아 센서를 부착하고 제어한다.
  - 앱 UI는 별도 설정창에서 `driver backend`, `launch command`, `working dir`, `startup wait`를 입력받는다.
- 완료 기준:
  - `manual`과 `external_command` 중 하나를 선택할 수 있다.
  - 외부 명령을 실행했을 때 앱이 ego vehicle을 준비하고, 종료 시 외부 프로세스도 정리된다.
  - Track / Practical 주요 실행 경로에서 동일한 방식으로 동작한다.

### 3. 결과 비교 표 또는 세션 요약 화면
- 상태: 예정
- 우선순위: 높음
- 목표:
  - 단일 결과 JSON 미리보기 수준을 넘어서, 최근 결과를 서로 비교할 수 있는 화면을 제공한다.
  - 발표 시연 시 "어떤 점이 좋아졌는가"를 바로 보여줄 수 있도록 한다.
- 구현 방향:
  - 최근 결과 2개 이상을 나란히 비교하는 간단한 표를 제공한다.
  - 비교 대상 항목:
    - score
    - completion
    - collision count
    - offroad ratio
    - lane score
    - wrong lane ratio
    - failure reason
  - 별도 팝업 창 또는 오른쪽 패널 확장 방식 중 UI를 덜 복잡하게 만드는 쪽을 선택한다.
- 완료 기준:
  - 사용자가 최근 결과 2개 이상을 비교할 수 있다.
  - Track / Intermediate / Practical 각각에서 동일한 기본 비교 흐름을 제공한다.

### 4. 난이도 조정 규칙 시각화
- 상태: 예정
- 우선순위: 중간
- 목표:
  - 다음 맵 또는 시나리오가 왜 그렇게 생성되었는지 사용자가 이해할 수 있도록 한다.
  - 특히 Practical Stage에서 `traffic`, `route length`, `junction focus` 변화 이유를 보여준다.
- 구현 방향:
  - "이전 결과 → 다음 환경" 변화 규칙을 짧은 설명문으로 보여주는 패널 또는 배지를 추가한다.
  - 예:
    - `collision이 많아서 traffic을 줄임`
    - `completion이 높아서 route length를 늘림`
    - `wrong_lane이 많아서 junction focus를 낮춤`
- 완료 기준:
  - Generate 또는 Generate + Run 이후, 앱에서 다음 환경 변화 이유를 바로 확인할 수 있다.
  - Track / Practical 중 최소 한 단계 이상에서 실제 적응 설명이 표시된다.

### 5. 시간이 남으면: 외부 주행 코드 인터페이스 구조 정리
- 상태: 예정
- 우선순위: 중간 이하
- 목표:
  - 외부 사용자가 자신의 ROS bridge / 자율주행 코드를 연결할 때 어떤 입력과 출력을 기대하는지 문서화한다.
  - 향후 확장 시 구조가 흔들리지 않도록 인터페이스 계약을 정리한다.
- 구현 방향:
  - 문서 또는 도식 형태로 정리
  - 포함할 내용:
    - app-owned ego vehicle 방식
    - role_name 기반 actor discovery
    - 외부 명령 실행 시 전달되는 환경 변수
    - 앱이 담당하는 범위
    - 외부 주행 코드가 담당하는 범위
- 완료 기준:
  - 외부 사용자가 문서만 보고도 연결 구조를 이해할 수 있다.
  - 최소한 Track / Practical에서 어떤 변수와 흐름을 쓰는지 설명되어 있다.

## 권장 구현 순서

1. 차량 모델 선택 기능
2. 주행 코드 선택 / 연결 기능
3. 결과 비교 표 또는 세션 요약 화면
4. 난이도 조정 규칙 시각화
5. 외부 주행 코드 인터페이스 구조 정리

## 구현 시 주의사항

- 현재 프로토타입은 이미 시연 가능한 수준까지 올라와 있으므로, 핵심 루프를 크게 흔드는 리팩터링은 피하는 편이 좋다.
- 메인 화면이 복잡해질 우려가 있는 항목은 별도 설정창 또는 팝업 창으로 분리한다.
- Track / Intermediate / Practical 세 단계를 모두 유지하면서 공통 구조를 확장하는 방식이 바람직하다.
- 외부 주행 코드 연결 기능은 "앱이 환경을 관리하고, 외부 코드가 차량 제어를 담당한다"는 현재 구조를 유지하는 선에서 구현한다.
