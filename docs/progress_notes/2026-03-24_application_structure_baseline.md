# 2026-03-24 Application Structure Baseline

## 1. 목적

맵 자동 생성 기능 구현에 집중하기 전에, 현재까지 만든 기능을 하나의 애플리케이션 구조로 정리한다.  
이번 단계의 목적은 완성형 GUI를 만드는 것이 아니라, **어떤 기능이 앱 책임이고 어떤 기능이 교체 가능한 입력 모듈인지**를 분명히 하는 것이다.

## 2. 현재 판단

현재 프로젝트에서 핵심은 다음과 같다.

1. 맵 생성
2. CARLA 맵 로딩
3. 차량 소환
4. 주행 결과 평가 및 저장
5. 다음 맵 재로딩

즉, 차량을 어떻게 조작하는지는 프로젝트 본체가 아니라 **교체 가능한 주행 모듈**에 가깝다.

## 3. 앱 책임

현재 애플리케이션이 직접 관리해야 하는 부분은 다음과 같이 정리한다.

### 3.1 맵 관련

- OpenDRIVE 맵 생성 요청
- 생성된 `.xodr` 파일 관리
- CARLA에 맵 로딩
- 시작점(`spawn_point`)과 목표점(`goal_point`) 관리

### 3.2 차량 관련

- 차량 모델 선택
- 차량 소환 및 재소환
- 맵 전환 시 기존 차량 정리

### 3.3 평가 관련

- 충돌 센서 부착
- 주행 경로 기록
- 중심선 기준 점수 산출
- 결과 JSON 저장

### 3.4 반복 실행 관련

- 현재 맵 다시 로드
- 다음 맵 생성/로드
- 맵 순환 세션 관리

## 4. 교체 가능한 입력 모듈

앱이 직접 소유하지 않고 교체 가능해야 하는 부분은 다음과 같다.

- 수동 주행
- CARLA 기본 autopilot
- 외부 자율주행 코드
- 향후 사용자가 제공하는 차량/주행 패키지

즉 최종 구조는 다음 개념으로 본다.

```text
AED Application
 ├─ Map Generator
 ├─ Map Loader
 ├─ Vehicle Spawn Manager
 ├─ Evaluation Engine
 └─ Driver Adapter
      ├─ Manual
      ├─ CARLA Autopilot
      └─ External Driving Code
```

## 5. 현재 코드 반영 상태

이번 단계에서 앱 구조를 위한 최소 코드 뼈대를 추가하였다.

### 5.1 앱 레벨 모델

- `scripts/app/models.py`

포함 내용:

- `CarlaRuntimeSpec`
- `MapGenerationSpec`
- `EvaluationSpec`
- `SessionUiSpec`
- `ApplicationSessionSpec`

### 5.2 앱 세션 팩토리

- `scripts/app/factory.py`

포함 내용:

- 현재 수동 맵 시퀀스 데모를 하나의 앱 세션으로 조합하는 함수
- 차량 프로필 / 주행 프로필 / 입력 바인딩 / 평가 기준을 앱 관점에서 묶는 로직

### 5.3 구조 요약 엔트리포인트

- `scripts/app/main.py`

이 파일은 현재 AED 애플리케이션이 어떤 구조로 동작하는지 요약 출력하는 진입점이다.

## 6. 현재 앱 상태의 해석

현재 상태는 “GUI가 없는 앱”에 가깝다.

- 백엔드 기능은 대부분 구현됨
- 주행 UI는 임시로 `pygame` 사용
- 장기적으로는 이 `pygame` 부분이 실제 앱 UI로 교체될 예정

즉 현재 구조는 다음처럼 이해하면 된다.

- **앱 본체는 이미 존재**
- **UI만 임시 상태**
- **핵심 남은 과제는 자동 맵 생성 알고리즘 고도화**

## 7. 앞으로의 우선순위

앱 구조를 지금 단계에서 너무 깊게 확장하지 않는다.

다음 우선순위는 다음과 같다.

1. 자동 맵 생성 파라미터 확장
2. 평가 결과와 다음 맵 생성 연결
3. 난이도 조절 로직 연결
4. 마지막에 앱 UI 정리

즉 지금은 앱 구조를 “틀” 수준으로만 잡아두고, 실제 개발 시간은 맵 생성 알고리즘에 집중하는 것이 맞다.
