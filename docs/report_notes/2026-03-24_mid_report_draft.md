## 1. 프로젝트 개요

본 프로젝트는 자율주행 차량 자체를 개발하는 것을 목표로 하지 않는다. 대신 자율주행 시스템을 실험하고 평가할 수 있는 주행 환경을 자동으로 생성하는 구조를 구현하는 것을 목표로 한다. 구체적으로는 CARLA 시뮬레이터를 기반으로 OpenDRIVE 형식의 도로 맵을 생성하고, 생성된 환경에서 차량이 주행한 결과를 기반으로 다음 환경을 다시 생성할 수 있는 구조를 설계하고 구현하고자 하였다.

프로젝트의 핵심은 환경 생성기(Environment Generator), 시뮬레이션 실행기(Simulation Runner), 평가 모듈(Evaluation Module), 난이도 조절 또는 다음 환경 생성 로직(Curriculum / Regeneration Logic)을 하나의 반복 가능한 루프로 구성하는 것이다. 즉 본 프로젝트는 자율주행 알고리즘 자체보다도, 자율주행 실험을 위한 환경 설계기(Environment Designer)를 만드는 데 초점을 둔다.

중간 단계에서는 전체 목표 중 가장 먼저 구현이 필요한 기반 기능인 OpenDRIVE 맵 생성, CARLA 로딩, 차량 소환, 반복적 맵 전환 구조를 우선적으로 구축하였다. 이를 통해 실제로 생성된 맵이 CARLA에서 동작하는지, 주행 가능한 코스를 반복적으로 불러올 수 있는지, 이후 평가 루프로 확장 가능한 구조가 성립하는지를 검증하였다.

## 2. 프로젝트 수행 배경 및 필요성

일반적인 자율주행 연구는 차량 제어, 센서 처리, 객체 인식, 주행 정책 학습에 집중하는 경우가 많다. 그러나 실제 학습 성능과 평가 결과는 어떤 환경에서 실험을 수행하는지에 따라 크게 달라진다. 환경이 지나치게 단순하면 일반화 성능이 떨어질 수 있고, 반대로 지나치게 복잡하면 초기 학습이나 반복 검증이 어려워진다.

따라서 자율주행 실험에 적합한 도로 환경을 자동으로 만들고, 현재 실험 목적에 맞는 구조로 다시 불러오며, 반복적으로 테스트 가능한 형태로 운영하는 기능은 매우 중요하다. 특히 CARLA와 OpenDRIVE를 함께 사용하면 표준적인 도로 형식을 기반으로 재현 가능한 실험 환경을 만들 수 있기 때문에, 졸업작품 수준에서도 충분히 의미 있는 결과물을 제시할 수 있다.

본 프로젝트는 이러한 점을 바탕으로 단순한 맵 제작을 넘어서, 실험용 주행 코스를 생성하고 반복적으로 로드하여 실험을 수행할 수 있는 기반 시스템을 목표로 한다.

## 3. 프로젝트 목표 및 현재 달성 현황

초기 계획서에서 설정한 전체 목표는 크게 다섯 가지였다.

1. OpenDRIVE 형식의 도로를 자동 생성하는 모듈 구현
2. 생성된 OpenDRIVE 맵을 CARLA에서 자동 로드하는 실행 구조 구현
3. 차량을 소환하여 주행 가능한 상태를 반복적으로 구성하는 구조 마련
4. 주행 결과를 평가하고 저장하는 모듈 구현
5. 평가 결과를 기반으로 다음 맵을 다시 생성하는 반복 루프 완성

현재 중간 단계까지 달성한 내용은 다음과 같다.

1. 직선, 곡선, 경기장형 트랙, 개방형 코스를 포함하는 OpenDRIVE 코스 생성 프로토타입을 구현하였다.
2. 생성된 `.xodr` 파일을 CARLA에 로드하여 실제 주행 가능한 도로 메시로 변환하는 기본 실행 기능을 확인하였다.
3. 로드된 맵 위에 CARLA 내장 차량을 소환하고, 수동 주행 또는 기본 실험용 주행 코드를 연결할 수 있는 상태를 구성하였다.
4. 두 개 이상의 맵을 준비한 뒤, 현재 맵을 다시 로드하거나 다음 맵으로 전환하고 차량을 재소환하는 반복 흐름을 시연 가능한 수준으로 구현하였다.
5. 차량 모델과 주행 방식 입력을 향후 외부 모듈로 연결할 수 있도록 구조를 분리하는 기반을 정리하였다.

반면 아직 최종적으로 완성되지 않은 부분은 다음과 같다.

1. 주행 결과의 정량 평가 지표를 완전하게 구현하는 부분
2. 평가 결과를 바탕으로 다음 환경의 난이도를 자동 조정하는 커리큘럼 로직
3. 외부 자율주행 코드와의 정식 연결 인터페이스

즉 현재까지는 전체 시스템 중 가장 중요한 기반 구간인 `맵 생성 -> 맵 로드 -> 차량 소환 -> 반복적 맵 전환`의 프로토타입을 확보한 상태이며, 이후 단계에서 평가 및 재생성 자동화를 이어갈 수 있는 토대를 마련하였다.

## 4. 현재 구현 범위

중간 단계에서의 구현 범위는 다음과 같이 정리할 수 있다.

### 4.1 포함된 구현 범위

- OpenDRIVE 기반 직선 도로 생성
- OpenDRIVE 기반 직선 + 곡선 조합 생성
- 경기장형 폐곡선 트랙 생성
- 시작점과 종료점이 분리된 작은 개방형 코스 생성
- CARLA에서 custom OpenDRIVE 맵 로드
- 생성된 맵 위 차량 소환 및 재소환
- 수동 주행 기반 맵 검증
- 다중 맵 전환 및 재로딩 구조
- 차량 모델과 주행 방식 입력 구조 분리의 기초 설계
- 작업 로그, 진행 기록, 스크린샷, 보고서용 문서 정리 체계 구축

### 4.2 현재 단계에서 아직 구현하지 않았거나 후속 구현 예정인 항목

- 실제 상용 수준의 자율주행 제어 알고리즘 구현
- 센서 기반 인지 시스템 구현
- 딥러닝 기반 에이전트 개발
- 복잡한 도시 전체 맵 생성
- 보행자, 신호등, 다중 교통 객체를 포함한 대규모 시나리오
- 평가 결과 기반 난이도 조절 및 다음 환경 자동 재생성 로직의 완성형 구현

특히 차량 주행 제어는 본 프로젝트의 핵심이 아니므로, 현재는 수동 주행 또는 CARLA 기본 기능을 이용한 검증 수준에 머무르고 있다. 반면 평가 결과 기반 난이도 조절과 다음 환경 재생성 로직은 본 프로젝트의 핵심 목표에 해당하지만, 중간 단계에서는 아직 프로토타입 기반을 마련하는 단계이므로 후속 구현 항목으로 남겨두었다.

## 5. 시스템 전체 구조

현재 프로젝트 시스템은 다음과 같은 네 가지 핵심 흐름으로 구성된다.

### 5.1 환경 생성 모듈

환경 생성 모듈은 OpenDRIVE XML 구조를 기반으로 직선, 곡선, 경기장형 트랙, 개방형 코스를 생성한다. 현재는 곡률, 길이, 차선 폭, 세그먼트 구성 등을 코드로 지정하여 `.xodr` 파일을 출력할 수 있다.

### 5.2 시뮬레이션 실행 모듈

시뮬레이션 실행 모듈은 생성된 `.xodr` 파일을 CARLA에 로드하고, 해당 맵 위에 차량을 소환하며, 사용자가 주행할 수 있는 상태를 구성한다. 또한 새로운 맵이 선택되었을 때 기존 맵을 정리하고 다음 맵을 다시 로딩하는 기능을 담당한다.

### 5.3 주행 입력 연결 구조

현재 단계에서는 주행을 수동 입력으로 수행하고 있으나, 구조적으로는 주행 방식이 외부에서 연결 가능한 형태로 정리되고 있다. 차량 모델 선택과 주행 방식은 각각 프로필 형태로 다룰 수 있도록 정리 중이며, 향후 외부 자율주행 코드가 들어오더라도 앱이 맵 로딩과 차량 소환을 담당한 뒤 주행 제어만 외부 모듈에 위임하는 구조로 확장할 수 있다.

### 5.4 평가 및 다음 환경 준비 구조

평가 모듈 자체는 아직 완성되지 않았지만, 이미 다음 맵을 준비하고 다시 로딩하는 반복 실행 구조는 구현되어 있다. 이는 향후 주행 결과를 JSON이나 CSV로 저장하고, 그 결과를 바탕으로 다음 코스를 다시 생성하는 루프로 자연스럽게 확장할 수 있는 기반이 된다.

## 6. 세부 구현 현황

### 6.1 CARLA 설치 및 연동

중간 단계에서 가장 먼저 수행한 작업은 CARLA 0.9.15를 Windows 환경에 설치하고, Python API와 연결하는 것이었다. 설치 과정에서 DirectX Runtime 문제와 Python 버전 호환성 문제가 있었으나 이를 해결한 뒤, Python 3.7 기반 환경에서 CARLA 서버 연결 및 맵 정보를 정상적으로 읽어올 수 있음을 확인하였다.

이 단계는 단순한 환경 세팅이 아니라 이후 모든 기능의 기반이 되었기 때문에 중간 구현에서 매우 중요한 의미를 가진다.

### 6.2 OpenDRIVE 맵 생성 기능

초기에는 가장 단순한 직선 도로 1개를 생성하는 `.xodr` 파일 생성기부터 시작하였다. 이후 직선과 곡선이 결합된 코스, 경기장형 폐곡선 트랙, 그리고 시작점과 도착점이 구분된 개방형 테스트 코스를 단계적으로 생성할 수 있도록 기능을 확장하였다.

특히 경기장형 트랙 생성은 “생성된 코스가 실제 트랙 형태를 가질 수 있는가”를 보여주는 중요한 중간 성과였으며, 개방형 코스 2개를 생성하여 반복 로딩 실험에 사용하는 단계까지 확장하였다.

### 6.3 CARLA 맵 로딩 기능

생성된 OpenDRIVE 파일은 `generate_opendrive_world()` 기반 로딩 구조를 통해 CARLA에서 실제 주행 가능한 도로로 변환되었다. 이 과정에서 텅 빈 공간 위에 도로만 표시되는 형태의 minimal map이 구성되었고, 이는 본 프로젝트가 목표로 하는 “코스 중심 실험 환경”과 정확히 부합한다.

즉 본 프로젝트는 건물, 신호등, 도시 배경 전체를 만드는 것이 아니라, 차량이 주행 가능한 도로 코스를 생성하는 데 초점을 두고 있다는 점이 중간 단계에서 명확해졌다.

### 6.4 차량 소환 및 주행 가능 상태 구성

맵 로딩 이후에는 CARLA 내장 차량을 이용하여 실제로 차량을 소환하고 주행 가능한 상태를 만드는 작업을 수행하였다. 초기에는 기본 autopilot을 이용해 주행 가능 여부를 검증하고자 하였으나, custom OpenDRIVE 트랙에서 주행 품질이 충분하지 않아 중간 검증 단계에서는 수동 주행 기반 시연으로 방향을 조정하였다.

이 과정에서 중요한 점은 차량 소환 자체는 앱이 담당해야 한다는 구조적 기준을 정리한 것이다. 즉 주행 방식이 수동이든, CARLA 기본 autopilot이든, 혹은 외부 자율주행 코드이든 간에, 맵 로딩과 차량 소환은 애플리케이션이 관리하는 것이 맞다는 기준을 확립하였다.

### 6.5 수동 주행 데모 구성

중간 단계에서는 사용자가 직접 키보드로 차량을 주행할 수 있도록 pygame 기반 수동 주행 화면을 구성하였다. 이를 통해 생성된 코스를 사람이 직접 확인하고, 실제로 맵이 주행 가능한 상태인지 검증할 수 있었다.

또한 서버 화면과 수동 주행 창의 역할을 분리하여, CARLA 서버는 맵 상태와 전체 환경 확인용으로 사용하고, 주행 조작 자체는 최소 기능의 수동 제어 창을 통해 수행하는 구조를 실험하였다.

### 6.6 맵 재로딩 및 다음 맵 전환 구조

중간 단계에서 가장 중요한 구현 성과 중 하나는 “현재 맵을 다시 로드하거나 다음 맵으로 전환하고 차량을 다시 소환하는 반복 구조”를 만든 것이다. 현재는 두 개의 작은 테스트 코스를 미리 생성하고, 사용자가 버튼 입력을 통해 현재 맵을 다시 로딩하거나 다음 맵으로 전환할 수 있다.

이는 향후 다음과 같은 전체 목표 구조의 핵심 토대가 된다.

1. 맵 생성
2. 맵 로딩
3. 차량 소환
4. 주행
5. 결과 저장
6. 다음 맵 생성 또는 선택
7. 다시 맵 로딩 및 차량 소환

즉, 중간 단계에서 이미 반복 실험의 최소 사이클이 구현된 셈이다.

### 6.7 차량 모델과 주행 방식 입력 구조 정리

중간 단계에서는 차량 모델과 주행 방식을 하나의 고정 로직으로 묶지 않고, 향후 앱에서 입력 가능한 요소로 분리하는 방향을 정리하였다. 현재 코드에서는 차량 프로필과 주행 프로필, 그리고 앱 입력 바인딩 개념을 두어 다음과 같은 구조를 염두에 두고 있다.

1. 차량 모델 입력
2. 주행 방식 입력
3. 앱이 차량을 소환
4. 외부 주행 모듈 또는 사용자 입력이 이미 소환된 차량을 제어

또한 경우에 따라 하나의 외부 코드가 차량 모델 정보와 주행 방식 정보를 함께 제공할 수도 있다는 점도 고려하였다. 그러나 이 경우에도 맵 로딩과 차량 소환의 주체는 앱이 유지하는 것이 바람직하다는 방향으로 정리하였다.

## 7. 구현 과정에서 발생한 문제와 해결 내용

### 7.1 CARLA 실행 환경 문제

초기 설치 단계에서 DirectX Runtime 오류가 발생하였다. 이는 필요한 런타임 구성요소를 설치한 뒤 해결하였다. 또한 Python 3.13 환경에서는 CARLA API와 호환성 문제가 있었기 때문에, CARLA 0.9.15에 맞는 Python 3.7 기반으로 실행 환경을 정리하였다.

### 7.2 OpenDRIVE 생성과 로딩 문제

처음에는 CARLA 맵 제작 자체가 매우 복잡할 수 있다는 우려가 있었으나, 전체 도시 맵 제작이 아닌 OpenDRIVE standalone 방식으로 범위를 제한함으로써 문제를 단순화하였다. 이로 인해 “텅 빈 공간 위에 도로만 있는 minimal environment”라는 현재 목표에 적합한 구현 전략을 확정할 수 있었다.

### 7.3 기본 autopilot의 한계

CARLA 기본 autopilot을 이용하여 custom OpenDRIVE 트랙 위를 주행시키는 실험을 수행하였으나, 벽에 붙거나 트랙을 안정적으로 따라가지 못하는 문제가 있었다. 이 문제를 해결하기 위해 임시 주행기와 센서 기반 실험도 일부 시도했지만, 프로젝트 본체가 자율주행 제어가 아니라 환경 생성이라는 점을 고려하여 중간 단계 시연은 수동 주행 기반으로 전환하였다.

이 변화는 단순한 포기가 아니라, 프로젝트 범위를 명확히 하고 본체 개발에 집중하기 위한 구조적 판단이었다.

### 7.4 작은 테스트 코스 전환 구조의 문제

두 개의 개방형 코스를 준비하여 반복 로딩 실험을 수행하는 과정에서, 차량이 코스 밖에서 소환되거나 잘못된 방향으로 스폰되는 문제가 있었다. 이를 해결하기 위해 각 맵 자체에 시작점 기준 spawn point 정보를 부여하고, 해당 맵의 진행 방향을 기준으로 소환 각도를 명시적으로 설정하는 방식으로 보완하였다.

이를 통해 맵마다 서로 다른 시작 위치와 방향을 가지더라도 차량을 일관된 방식으로 소환할 수 있는 구조를 마련하였다.

## 8. 중간 결과 정리

현재까지의 구현 결과를 요약하면 다음과 같다.

1. CARLA 설치 및 Python API 연동 완료
2. OpenDRIVE 기반 직선 도로 생성 성공
3. OpenDRIVE 기반 직선 + 곡선 코스 생성 성공
4. 경기장형 폐곡선 트랙 생성 및 로딩 성공
5. 개방형 테스트 코스 2개 생성 및 반복 로딩 구조 구현
6. 맵 로딩 후 차량 소환 성공
7. 수동 주행 기반 시연 가능 상태 확보
8. 다음 맵 로딩 및 차량 재소환 반복 구조 구현
9. 차량 모델/주행 방식 입력 구조 분리의 기초 설계 완료

따라서 중간 단계 기준으로 “생성된 맵을 CARLA에 로드하고, 차량을 올려 주행 가능한 상태로 만든 뒤, 새로운 맵을 다시 불러오는 반복 구조”는 기본 프로토타입 수준에서 검증된 상태라고 볼 수 있다.

## 9. 향후 개발 계획

중간 단계 이후에는 현재 구현된 기반을 바탕으로 다음 작업을 수행할 계획이다.

### 9.1 평가 지표 구현

주행 결과를 기반으로 완주 여부, 충돌 여부, 주행 시간, 중심선 이탈 정도, 평균 속도 등의 지표를 계산하는 평가 모듈을 구현할 계획이다.

### 9.2 결과 저장 구조 확장

각 맵에서 수행된 주행 결과를 JSON 또는 CSV 형태로 저장하고, 이후 실험 반복 시 누적 결과를 비교 분석할 수 있도록 할 계획이다.

### 9.3 다음 환경 자동 생성 로직 구현

현재는 버튼 입력을 통해 다음 맵을 직접 불러오는 방식이지만, 이후에는 평가 결과를 바탕으로 다음 맵 생성 파라미터를 자동 조정하는 구조를 구현할 예정이다.

### 9.4 외부 주행 모듈 연결 구조 정리

현재는 수동 주행 중심으로 검증하고 있으나, 향후에는 외부 자율주행 코드가 이미 소환된 차량을 받아 제어할 수 있도록 인터페이스를 보다 명확히 정리할 계획이다.

## 10. 결론

본 프로젝트는 현재 중간 단계에서 가장 중요한 기반 기능의 프로토타입을 확보하였다. 즉 OpenDRIVE 기반 코스를 생성하고, 이를 CARLA에 로딩하며, 차량을 소환하여 실제로 주행 가능한 상태를 만들고, 이후 새로운 맵을 다시 로딩하여 반복 실험 가능한 구조를 확인하였다.

비록 평가 지표 및 커리큘럼 기반 난이도 조절은 아직 완성 단계에 이르지 않았지만, 본 프로젝트의 핵심 토대인 환경 생성기와 시뮬레이션 실행기의 기본 구조는 확보되었다. 또한 차량 제어 자체는 프로젝트 본체가 아니라 외부 입력 또는 교체 가능한 모듈로 취급한다는 구조적 기준을 정리함으로써, 향후 확장 가능한 형태의 시스템 아키텍처도 함께 마련하였다.

따라서 현재 시점의 중간 결과는 단순한 아이디어 수준을 넘어, 실제로 작동하는 OpenDRIVE 기반 CARLA 환경 생성 및 반복 실행 프로토타입을 확보한 단계로 평가할 수 있다.
